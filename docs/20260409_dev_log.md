# 🏛️ SpineDoc 开发日志 - 2026/04/12

## 1. Windows 内存优化行动 (Operation: Memory Rescue)
- **挑战：** 在 32GB RAM 的 Windows 宿主机上，使用 `ProcessPoolExecutor` 并行渲染大型 PDF 时触发 `malloc failed` 内存溢出。
- **根因：** 
    - Windows 的多进程 spawn 机制导致每个子进程重复加载 PDF 索引与 PyTorch 环境，内存开销呈指数级增长。
    - 频繁 `fitz.open` 同一大文件导致操作系统读缓存碎片化。
- **重构方案：**
    - **单句柄渲染：** 废弃多进程渲染，改用主进程单句柄顺序渲染图片。
    - **算力分层并行：** 渲染设为串行（保障内存安全），OCR 识别维持异步并发（保障处理速度）。
    - **自愈机制：** 在渲染函数中引入 `gc.collect()` 紧急回收与降级重试逻辑。

## 2. 脊梁集成进度 (Spine Integration)
- **叶子分片：** 成功提取 Leaf Nodes，确立了物理区间的绝对隔离。
- **阈值调优：** 将分片敏感度提升至 0.82，成功捕获微小的语义转折点。
- **环境一致性：** 强制宿主机运行在 CPU 模式，彻底解决了 CUDA DLL 冲突导致的 `WinError 127`。

## 3. 三段式 OCR 精度重构 (Operation: Precision Boost)
- **问题：** 正文页 OCR 识别精度不足，复杂页面出现漏字或乱码。
- **根因：**
    - **过度下采样：** 原逻辑将图像强制限制在 1120px 维度，导致密集文本细节丢失。
    - **Token 瓶颈：** `max_tokens` 限制在 512，导致长 Markdown 输出被强行截断。
    - **长宽比失衡：** 单页全景识别无法充分利用 GLM-OCR 的长文窗口优势。
- **重构方案：**
    - **三段式切分：** 在 `GLMWorker` 中实现垂直三段式（上、中、下）切分识别逻辑，每段保留 150px 重叠区域以防文字截断。
    - **分辨率提升：** 将分段后的图像 `max_dim` 提升至 1536px，大幅增加像素密度。
    - **输出扩容：** 将 `max_tokens` 提升至 1024，允许生成更完整的 Markdown 结构。
    - **并发控制：** 引入 `asyncio.Semaphore` 限制分段请求并发，防止压垮本地 vLLM 实例。

## 4. 架构范式转换：从“全能幻觉”回归“确定性工装” (Operation: Deterministic Spine)
- **挑战：** GLM-OCR (0.9B) 作为一个自回归 VLM，在低密度或结构化页面上表现出不可控的“对话式幻觉”和英文污染，破坏了 SpineDoc 的逻辑审计严肃性。
- **根因分析：**
    - **非确定性风险：** VLM 试图“理解”而非“识别”，导致输出包含非原文的解释词。
    - **算力浪费：** 对简单的文字识别动用复杂的 Decoder 架构，不仅增加了显存负担，还引入了复读噪声。
- **重构决策：** 彻底解耦“感知”与“理解”，构建模块化感知链。
    - **文字与布局 (The Sniffer)：** 引入 PaddleOCR (RapidOCR)。采用判别式架构，显存占用压低至 <1GB，实现 100% 确定性的字符提取。
    - **公式专项 (The Specialist)：** 引入 LaTeX-OCR (pix2tex)。采用“外科手术式”切片识别，强制运行在 CPU 模式（显存占用 0MB），确保数学公式的 LaTeX 语法精准无误。
    - **缝合逻辑 (The Assembler)：** 在 `BodyAlchemist` 层面通过坐标对齐，将分散的文字块与公式块缝合为 Markdown 脊梁。
- **安全纪律：** 严守“不破坏现有生产路径”原则。在 `PaddleWorker` 逻辑未通过 100% 回归测试前，保留 `GLMWorker` 代码作为热备方案。

## 5. 确定性感知链实战验证成功 (The Lab Victory)
- **实战结果：** `tests/debug_tools/test_deterministic_pipeline.py` 在真实样本 `ceshi_ocr/image.png` 上运行成功。
- **关键突破：**
    - **幻觉归零：** 彻底杜绝了 GLM-OCR (0.9B) 产生的 `"Looking closely..."` 等英文污染。
    - **公式复原：** 成功捕获并精修了复杂的数学公式，LaTeX 语法精准，且不占用 GPU 显存（强制 CPU 运行）。
    - **资源极致压缩：** GPU 显存占用从 4GB+ 压低至 <1GB，为大规模 RAG 审计腾出了核心算力空间。
- **决策：** 全面弃用基于自回归 VLM 的全量 OCR 方案，重构 `BodyAlchemist` 流水线，采用“嗅探-精修”的确定性架构。

## 9. 逻辑确定性增强：守护者模式与 TOC Bypass (Operation: Zero Gap)
- **挑战：** 
    - **目录前内容丢失：** 封面、版权页等内容在 RAG 中失联。
    - **逻辑对齐漂移：** 自动探测 Offset 的不可靠性导致正文页码错误。
    - **层级污染：** 引入 Level 0 守护节点可能导致正文逻辑整体下沉。
- **重构决策 (V48.0 - V48.2)：**
    - **用户确权 Offset (Deterministic Offset)：** 显式引入用户输入参数。若用户提供 Offset，则废弃所有启发式探测逻辑，实现 100% 确定性对齐。
    - **并列守护者 (Parallel Guardians)：** 合并“前言”与“序言”为单一节点 `[Introductory Material]`。
    - **层级平衡：** 守护节点同样设为 `Level 1`，与正文一级标题处于**并列根节点**地位，防止树形结构发生非预期下沉。
    - **TOC OCR Bypass 彻底修复 (V48.2)：** 修复了只跳过区间首尾两页的 Bug，在进入 OCR 前将坐标对 `[start, end]` 全量展开为物理页码列表。此举彻底屏蔽了目录页的无效 OCR 算力消耗。
- **阶段性里程碑 (Milestone 1 Reached)：**
    - 我们成功完成了 **RAG 的逻辑原材料 (Logical Raw Materials)** 准备阶段。
    - 文档不仅被物理切碎，而是通过 ISR (隐性脊梁) 被重塑为一棵携带全量语义指纹、物理位置绝对对齐、100% 零信息丢失的“知识树”。
- **状态：** 代码注入与全链路审计完成，准备同步至 Gitee。
